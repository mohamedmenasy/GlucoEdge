# Android app: on-device trend classification against a replayed CGM stream

Date: 2026-07-05
Status: approved design

## Purpose

Third sub-project of GlucoEdge (after the training pipeline and the LiteRT
conversion): a Kotlin + Jetpack Compose Android app that runs the converted
glucose trend classifier **on-device** via LiteRT's **CompiledModel API**
against a **replayed** CGM CSV stream. No live sensor. **No network calls
for inference, ever.** This is a portfolio demonstration of production
on-device ML deployment — explicitly not a medical device.

It also closes the one measurement the conversion phase deferred: real
Android inference latency for both the float and INT8 models (the
conversion results doc reported only a dev-machine CPU proxy).

## Decisions (settled with the user)

1. **Models: bundle both, float default.** Both `.tflite` artifacts ship as
   app assets (~29 KB total). Float drives the UI by default (the
   conversion phase's documented recommendation); a UI toggle switches to
   INT8. The app measures on-device latency for whichever model is active.
2. **Replay data: real extract + synthetic fallback.** A committed Python
   script extracts one contiguous segment of one weinstock patient from the
   *local* GlucoBench clone into `android/app/src/main/assets/real_trace.csv`
   (**gitignored** — GlucoBench data is never committed). A small
   synthetic-but-plausible trace generated by our own code
   (`synthetic_trace.csv`) **is** committed, so the app and CI always build
   without GlucoBench. At runtime the app prefers `real_trace.csv` if
   present, else falls back to the synthetic trace, and labels which one is
   playing in the UI.
3. **Model assets are committed.** Scoped `.gitignore` exception for
   `android/app/src/main/assets/*.tflite` only (these are our own trained
   weights on public data — the never-vendor rule covers GlucoBench data,
   not our artifacts). `results/` stays ignored. Each committed model's
   provenance (producing commit, seed, sha256) is recorded in
   `android/app/src/main/assets/MODELS.md`.
4. **Structure: Approach A** — single-module Compose app at `android/`
   inside this repo. Clear package boundaries instead of multi-module
   ceremony. No DI framework, no navigation library (one screen).
5. **UI: one screen + stats panel** (details below).

## Repo layout

```
android/
├── settings.gradle.kts, gradle wrapper, gradle/libs.versions.toml
└── app/
    ├── src/main/kotlin/com/glucoedge/app/
    │   ├── replay/      # CsvTraceLoader, ReplayEngine, WindowBuffer
    │   ├── inference/   # TrendClassifier, LatencyMeter
    │   └── ui/          # MainScreen (Compose), MainViewModel, theme
    ├── src/main/assets/ # trend_float.tflite, trend_int8.tflite,
    │                    # synthetic_trace.csv, MODELS.md  (committed)
    │                    # real_trace.csv                  (gitignored)
    ├── src/test/        # JVM unit tests (run in CI)
    └── src/androidTest/ # golden-vector parity test (local emulator)
```

Kotlin DSL, version catalog, Jetpack Compose + Material 3, `ViewModel` +
`StateFlow`, coroutines. The exact `com.google.ai.edge.litert` artifact and
version, minSdk, and compile/target SDK are pinned at plan time from
current LiteRT documentation (the CompiledModel API is the required
inference surface; the artifact choice follows from that).

**No-network guarantee, structurally:** the manifest declares **no**
`INTERNET` permission, so the OS forbids network I/O app-wide — a stronger
claim than "the code doesn't call the network." A JVM test asserts the
merged manifest contains no `INTERNET` permission so a dependency can't
smuggle one in.

## Data flow

```
CSV (real_trace.csv if present, else synthetic_trace.csv)
  → CsvTraceLoader   parse (time, gl) rows; validate 5-min cadence;
                     count + skip malformed rows
  → ReplayEngine     coroutine clock, one reading per tick;
                     play/pause/speed; decoupled from wall time
  → WindowBuffer     sliding window of 12 readings; RESETS on any
                     time jump > 5 min between consecutive readings
                     (mirrors training's id_segment rule: never let a
                     window span a data gap)
  → TrendClassifier  12 raw mg/dL values → (1,1,12) → CompiledModel →
                     5 logits → softmax → probabilities + argmax
  → UiState (StateFlow) → single Compose screen
```

Replay never blocks on inference; each new complete window triggers one
inference on a background dispatcher. Speeds: 1× / 4× / 16× = one reading
per 5 s / 1.25 s / 0.31 s (replay seconds, not the real 5-minute cadence).

## Inference layer

`TrendClassifier` wraps one LiteRT `CompiledModel` per model file; the
float/INT8 toggle disposes the current instance and loads the other
(models are tiny; reload cost is negligible).

- **Float path:** raw mg/dL values in, logits out, softmax in Kotlin.
- **INT8 path:** the model has true int8 input/output boundaries. Kotlin
  input quantization replicates the *corrected* Python benchmark exactly:
  `round(x/scale) + zeroPoint`, **clipped** to [-128, 127]; output
  dequantized as `(q - zeroPoint) * scale`. Scale/zero-point are read from
  the model at runtime, never hardcoded. If the CompiledModel API surface
  does not expose quantization parameters, the plan may read them once at
  load time via the Interpreter API — the hot inference path stays
  CompiledModel either way.
- `LatencyMeter`: wall time per inference, rolling last-100 window,
  mean/median/p95 per model. These are the deferred "real on-device"
  numbers. Honesty note: until a physical device is used, measurements come
  from the emulator; the UI stats panel and README must label them
  "emulator" vs "device" (detected via `Build.FINGERPRINT`/`Build.MODEL`
  heuristics or simply labeled by how the test was run — plan decides, but
  the label must exist).

## Golden-vector parity test (correctness anchor)

A committed Python script (`conversion/export_golden_vectors.py` or
similar; plan decides the path) runs at least 16 windows **from the
synthetic trace only** (never GlucoBench data), chosen so all five
predicted classes appear and at least one window contains values above the
INT8 ~240 mg/dL ceiling, through both committed `.tflite` files using
the existing corrected benchmark code path, and writes
`golden_vectors.json` (inputs, float logits, int8 raw outputs, expected
classes, model sha256s) into `androidTest` resources. The instrumented
test:

1. asserts each bundled asset's sha256 matches both `MODELS.md` and the
   goldens file (regenerating a model without regenerating goldens fails
   loudly);
2. feeds the same windows through `TrendClassifier` on the emulator/device
   and asserts outputs match — exact for INT8, tolerance ≤ 1e-5 for float.

This proves the Kotlin input layout, quantization handling, and output
decoding are faithful to what the paper and conversion record report.

## UI (single screen, top to bottom)

- Line chart of the last hour's 12 readings — plain Compose `Canvas`, no
  chart library.
- Current glucose value (mg/dL) + which trace is playing (real/synthetic).
- Predicted trend arrow + class name, captioned **"predicted trend, next
  15 min"** (the label is a forecast, and the UI must say so).
- Per-class confidence: 5 labeled bars.
- Stats panel: active model (float/INT8), inference count, mean/p95
  latency, malformed-row count, emulator/device label.
- Replay controls: play/pause, speed (1×/4×/16×).
- Model toggle: float ↔ INT8.
- Persistent footer: *"Portfolio demo on public research data — not a
  medical device, not treatment guidance."*

## Error handling

- Malformed CSV rows: skipped, counted, count surfaced in stats panel.
- Zero valid readings: explicit error state, not an empty chart.
- Inference failure: non-crashing error banner (should be unreachable with
  committed assets, but the app must not crash).
- Window reset after a time jump: brief "window rebuffering…" indicator
  while the buffer refills (the arrow pausing has a visible reason).

## Testing

- **JVM unit tests (CI):** CSV parsing (incl. malformed rows), window
  sliding + reset-on-gap, quantize/dequantize math against hand-computed
  values **including inputs above the ~240 mg/dL representable ceiling**
  (the exact bug class the conversion-phase review caught — the clip must
  be tested), replay timing on a virtual clock, no-INTERNET-permission
  manifest assertion.
- **Instrumented tests (local, documented command):** golden-vector parity
  for both models on an emulator or device. Not run in CI.
- **CI:** existing GitHub Actions workflow gains a second job: JDK 17 +
  Android SDK setup, `:app:assembleDebug :app:testDebugUnitTest`. The
  pytest job is unchanged. README states plainly that parity tests are a
  local gate.

## Out of scope (YAGNI)

Live sensor input, persistence/history, settings screen, LiteRT-LM
explanation layer (separate stretch sub-project), Play-store packaging,
physical-device benchmarking automation, accessibility polish beyond
Compose defaults.

## Constraints carried from the project brief

- Public GlucoBench data only; **no GlucoBench data committed to the repo**
  (the real trace stays gitignored; goldens and the committed trace are
  synthetic).
- No proprietary code, data schemas, or algorithms from prior
  FDA-regulated work.
- No network calls for inference, ever (enforced by absent permission).
- Nothing in code, comments, or UI strings may imply clinical validity or
  treatment guidance.

## Open items pinned at plan time (not open questions)

- Exact LiteRT Maven artifact + version exposing the CompiledModel Kotlin
  API; minSdk/compileSdk/targetSdk to match.
- Whether CompiledModel exposes I/O quantization params directly or the
  loader reads them via a one-shot Interpreter.
- AGP/Gradle/Kotlin versions (current stable at plan time).
- Synthetic trace generator shape (plausible mg/dL dynamics covering all
  five classes, including values above 240 so INT8 saturation is visible
  in the demo and covered by goldens).

## Implementation notes (recorded deviations)

Inference runs on the `ViewModel`'s main-dispatcher `collect` rather than a
background dispatcher; this is plan-sanctioned (the plan doesn't mandate a
background dispatcher), and moot in practice since the ~1 ms per-model
latency makes blocking the UI thread a non-issue — the reported latency
numbers therefore share the UI thread with everything else. The float
parity tolerance shipped at 1e-5 (the tighter of the two the spec allowed):
all 20 golden vectors pass at that tolerance on the emulator, so the 1e-4
fallback was never needed.
